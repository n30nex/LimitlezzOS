1. Vision & Design Philosophy
limitlezzOS is a purpose-built mesh operating environment for the LilyGO T-Deck. Its goal is to make mesh networking accessible to everyone — not just technically proficient users. Think of it as the 'just works' mesh device. The target user heard about mesh networks after a hurricane, or wants to stay in touch on a hiking trail, or is at a festival without cell coverage. They should be on the mesh in under one minute with zero technical knowledge required.

Core Principles — These Are Non-Negotiable
Mesh for the masses — every decision prioritizes simplicity over power-user features
Plain language everywhere — no technical jargon in the UI. 'Weak signal' not 'SF12 BW125'. 'Sending slowed — radio cooling down' not 'duty cycle limit reached'. 'No one nearby yet' not 'no nodes in range'
Ultra-lightweight kernel — the OS shell consumes minimum RAM and flash so the radio stacks and app VM get the headroom they need
No app touches the kernel — all third-party code runs inside a Lua VM. No native execution, no direct hardware access from apps, ever
One UI, two meshes — Meshtastic and MeshCore are unified behind a single inbox. The network tag is always visible but never intrusive
Foreground-only apps — only preloaded system services run in the background. User-installed apps run only when active
Zero runtime assets for the OS shell — the OS draws everything as primitives and baked font tables. No PNG files, no bitmap loading for the OS itself. Apps may use assets from their SD sandbox

3. Radio Architecture — TDM Dual-Network
3.1 The Core Constraint
The SX1262 is a single half-duplex transceiver. Meshtastic and MeshCore cannot use the radio simultaneously. Time-division multiplexing (TDM) shares the radio between both protocol stacks. This is the foundational architectural decision the entire OS is built around.

3.2 RF Parameter Alignment — Critical
TDM is only efficient if both networks share identical LoRa modem parameters. When frequency, bandwidth, spreading factor, and coding rate all match, switching between networks requires only a sync word register write — no PLL recalibration, sub-millisecond cost. This is what makes TDM practical.

RF Parameter
TDM Requirement
Frequency
MUST match on both networks
Bandwidth
MUST match on both networks
Spreading Factor
MUST match on both networks
Coding Rate
MUST match on both networks
Sync Word
DIFFERENT per network — hardware-filters packets at SX1262 level
Preamble Length
Recommended match to reduce settling time

3.3 TDM Scheduler
The scheduler lives in the Radio HAL layer. It is the sole owner of the SX1262. Neither protocol stack accesses hardware directly.

Default slice: 50/50 adaptive — widens toward the network with queued TX or recent RX activity
Slice length baseline: ~300ms — tune per spreading factor. Higher SF = longer airtime = longer slices
Slice transitions are IRQ-gated: a detected preamble extends the current slice until RX_DONE fires. Never switch mid-packet
TX is queued per-network. Arbiter drains queues respecting CSMA and regional duty-cycle limits
Duty-cycle accounting is SUMMED across both networks. The arbiter enforces the aggregate limit
Timer fires the intent to switch. IRQ state gates whether the switch actually happens now or after the in-flight packet completes

3.4 Layer Ownership
Layer
Responsibility
Radio HAL
Owns SX1262 exclusively. TDM scheduler lives here
Protocol Stacks
Meshtastic stack + MeshCore stack. See virtual radio interface, not hardware
Arbiter
Mediates between stacks and the HAL. Manages duty cycle, queuing, CSMA
Messaging Service
Receives parsed packets from the arbiter. Background system service
App SDK mesh API
Apps call mesh.send() / mesh.onReceive(). Goes through arbiter. Never reaches HAL

4. Kernel Architecture
4.1 Philosophy — Ultra-Lightweight
The kernel is not a general-purpose RTOS. It is a minimal event loop with cooperative task scheduling. FreeRTOS is used underneath only for its queue and timer primitives — not for OS-style task management. Everything that can be cut is cut. The goal is maximum RAM headroom for the Lua VM and dual protocol stacks.

No dynamic memory allocation in the kernel after boot — static allocation wherever possible
Cooperative scheduling — tasks yield. No preemptive context switching at the kernel layer
RAM budget enforced at compile time — kernel + both mesh stacks must fit within a defined SRAM ceiling. PSRAM is reserved for Lua VM heap and app assets

This is enforced at the OS level, not advisory.

4.4 Display Rendering
The OS shell uses LVGL with aggressive stripping via lv_conf.h. Disable all unused widgets, animation engine, theme system, extra font formats, and image decoders. The OS shell has zero runtime assets — everything is primitives and baked font tables compiled into flash. LVGL is configured once and never loads files for OS chrome rendering.

Strip: animation engine, theme system, PNG/JPEG decoder, unused widgets (calendars, charts, sliders, spinners, on-screen keyboard)
Keep: display flush pipeline, basic draw primitives, input device abstraction (trackball + keyboard), task handler / tick system
Display updates use dirty-region rendering — only changed regions are pushed to the ILI9341 over SPI
Status bar redraws independently from the message list. A new message appends without touching the status bar

6.4 Reply Routing — Must Be Unambiguous
A conversation is bound to the network it arrived on. Hitting send always routes to that network — no silent network selection ever. A contact on both networks appears as two separate entries. The user explicitly chooses which thread to reply in.

6.5 Network Disable Behavior
Disabling a network stops TDM slicing to that profile — immediately improves the other network's latency and throughput. Surface this benefit in settings: 'Disabling a network gives the other network more radio time'
Conversations from a disabled network are grayed out, not deleted
Re-enabling restores full history and resumes TDM slicing

6.6 Status Bar
Drawn entirely as primitives — no image assets. Left to right:
Signal bars — 4 rectangles of increasing height. Green = good, yellow = okay, red = weak, gray = none. If both networks active: two stacked mini-bar sets, one cyan, one amber
Display name — small monospace, always visible
Battery icon — outline rectangle + fill level rectangle + optional % text if enabled in settings + lightning bolt primitive when charging
DND glyph — small moon shape drawn as primitive, visible when DND is active

Battery % is OFF by default. User enables in settings. RSSI/SNR values are hidden by default — accessible via Settings > Diagnostic Mode toggle which flips the whole OS into 'show numbers' mode for power users.

8. Hardware Feedback System

One background system service owns all three hardware feedback outputs: status LED, buzzer, and keyboard backlight. Nothing else in the OS drives these outputs directly. All requests — from system services and from apps with notification permission — go through the Feedback Manager. It resolves conflicts via a priority queue and applies current DND and power state.

9. App Platform — Lua Sandbox
9.1 Runtime
Third-party apps run as Lua 5.4 scripts inside an embedded VM. The VM is the privilege boundary. Apps cannot express native hardware operations because the VM does not expose them. This is the iOS-style isolation model — enforced structurally, not just by policy.

Lua 5.4 — small footprint, designed for embedding, well-suited to ESP32-S3 PSRAM heap
Per-app memory cap — enforced via Lua allocator hook. App exceeding limit is killed cleanly
Instruction-count watchdog — hook fires every N instructions, yields to OS event loop, kills app if it has not yielded within timeout window. A spinning app cannot starve the radio scheduler
All crashes caught via pcall — a faulting script throws to the launcher, never to the OS. App crash = launcher shows error message, returns to home screen. Radio keeps running
Foreground-only — app is terminated when user navigates away. State saved to SD sandbox before termination. Restored on relaunch

9.2 App Package Format
Each app is a folder on the SD card: /apps/<appname>/

9.5 Security Properties — What an App Can NEVER Do
Write to arbitrary memory addresses
Directly access or reconfigure the SX1262
Change TDM scheduler parameters
Read mesh identity keys or node credentials
Access another app's data directory
Run while backgrounded
Exceed its memory allocation
Block the OS event loop indefinitely
Inject raw LoRa packets — mesh access is mediated by the arbiter only
Control the LED, buzzer, or keyboard backlight directly — all feedback goes through Feedback Manager

9.6 Filesystem Jail
Storage API refuses any path containing '..' or an absolute path
An app that attempts a filesystem escape is killed and flagged in the launcher
OS config, mesh identity keys, and other apps' data are invisible to apps

10.4 Developer Experience Goal

11. Over-The-Air OS Updates
ESP32-S3 supports OTA firmware updates natively via ESP-IDF's esp_ota API. No USB flashing required for updates.

11.1 How It Works
Flash partition layout: OTA_0 + OTA_1 (two full firmware slots) + OTA data partition (tracks active slot)
Update process: download new firmware binary over WiFi into inactive partition → verify SHA256 → write OTA data partition to point to new slot → reboot → new firmware runs
If new firmware fails to boot: OTA data partition rolls back to previous slot automatically

11.2 Integration with App Store
App Store service checks a firmware update channel alongside the app catalog
User sees: 'OS update available — version X.X. Download and install?' Plain language, explicit confirmation required
During OTA install: LED fast white pulse, screen shows progress, keyboard and mesh remain available for receiving messages
Partition layout must be defined correctly BEFORE the firmware grows large — this is a flash budget decision to make at project start

NOTE: Define the partition table (partitions.csv) before significant firmware development. Two OTA partitions + OTA data + app data must all fit within 16MB flash. Plan this early.

12. Emergency Beacon
A dedicated distress broadcast feature. Designed for the disaster/outdoor use case that is one of limitlezzOS's core markets.

Activation: dedicated key combo (e.g. hold two specific keys for 3 seconds) — hard enough to trigger accidentally in a pocket, trivial when intentional
Action: broadcasts a distress message on BOTH networks simultaneously, bypassing normal TDM scheduling — emergency traffic gets immediate airtime
Feedback: LED rapid red/white alternate + continuous SOS buzzer pattern until dismissed by user
Incoming emergency beacon from another node: always wakes screen and sounds buzzer even in DND mode — safety overrides silence
Plain language: 'Send emergency alert to everyone nearby?' confirmation before firing

14. Build Roadmap
IMPORTANT STAGING NOTE — READ FIRST

Do not attempt to build everything at once. The correct strategy is to build a fully working, polished OS on Meshtastic first — single network, single stack, no TDM complexity. Get that rock solid. Then add MeshCore as a second network layer on top of a proven foundation. This approach means bugs are always isolated: if something breaks after adding MeshCore, you know exactly where to look. Trying to integrate both stacks simultaneously while building the kernel and UI backend creates a debugging nightmare where nothing can be trusted.

NOTE: Build the BASE OS on Meshtastic only. Add MeshCore support in Stage 2. Do not begin Stage 2 until Stage 1 is stable, tested, and dogfooded on real hardware.

Stage 1 — Base OS (Meshtastic Only)
Goal: a fully working limitlezzOS device on Meshtastic. Everything the end user experiences should be complete and polished at the end of Stage 1. MeshCore is simply not present yet — the UI shows one network, the radio runs one stack, there is no TDM scheduler yet (single stack owns the radio directly). This is a real, shippable product at the end of Stage 1.

15. Open Questions — Resolve Before Building
RF alignment: Verify Meshtastic and MeshCore can be configured to identical BW/SF/CR on the same frequency in all target regions before committing to TDM. Check both codebases.
MeshCore radio abstraction: Confirm MeshCore's codebase exposes the radio cleanly enough to run as a stack instance rather than owning hardware directly.
Lua footprint: Measure Lua 5.4 base RAM cost on ESP32-S3 against the SRAM budget. If too large, evaluate eLua or a custom minimal interpreter.
Partition table: Define partitions.csv before significant firmware growth. Two OTA slots + OTA data + app/config storage must fit within 16MB flash.
App state serialization: Define the exact SDK API for an app to save and restore state. Decide if this is mandatory (enforced by launcher) or advisory.
Catalog trust model: SHA256 hashes prevent corruption but not malicious apps. Define signing requirement for catalog listing before the store goes public.
